# Спецификации прецедентов

## UC-01. Регистрация пользователя
**Актор:** пользователь  
**Предусловия:** пользователь не авторизован  
**Результат:** создана новая учётная запись и профиль

**Основной поток:**
1. Пользователь открывает экран регистрации.
2. Вводит email, пароль и имя.
3. Приложение отправляет запрос `POST /api/auth/register`.
4. Backend валидирует данные и создаёт учётную запись.
5. Backend возвращает JWT и данные пользователя.
6. Mobile сохраняет токен и переводит пользователя в основной интерфейс.

**Альтернативы:**
- email уже существует — приложение показывает сообщение об ошибке;
- пароль не проходит проверку — ввод требуется повторить.

## UC-02. Авторизация пользователя
**Актор:** пользователь  
**Предусловия:** учётная запись уже создана  
**Результат:** пользователь получает доступ к защищённым разделам

**Основной поток:**
1. Пользователь вводит email и пароль.
2. Приложение отправляет запрос `POST /api/auth/login`.
3. Backend проверяет учётные данные.
4. Backend возвращает JWT.
5. Mobile сохраняет токен в локальное хранилище.
6. Открывается dashboard.

**Альтернативы:**
- неверный пароль — вход отклоняется;
- сервер недоступен — пользователь получает сообщение об ошибке сети.

## UC-03. Редактирование профиля
**Актор:** пользователь  
**Предусловия:** пользователь авторизован  
**Результат:** профиль обновлён и использован для расчёта рекомендаций

**Основной поток:**
1. Пользователь открывает экран профиля.
2. Изменяет параметры тела, цель и уровень активности.
3. Приложение отправляет запрос `PUT /api/profile/me`.
4. Backend сохраняет данные.
5. Профиль обновляется в mobile и на dashboard.

## UC-04. Управление блюдами
**Актор:** пользователь  
**Предусловия:** пользователь авторизован  
**Результат:** список блюд актуален

**Основной поток:**
1. Пользователь открывает раздел питания.
2. Просматривает список блюд.
3. При необходимости создаёт, редактирует, удаляет или ищет блюдо.
4. Приложение обращается к `/api/meals` и `/api/meals/search`.
5. Backend выполняет CRUD-операции и возвращает обновлённый список.

## UC-05. Управление тренировками
**Актор:** пользователь  
**Предусловия:** пользователь авторизован  
**Результат:** пользователь управляет тренировочным планом

**Основной поток:**
1. Пользователь открывает раздел тренировок.
2. Просматривает список тренировок.
3. При необходимости создаёт, редактирует, удаляет или ищет тренировку.
4. Приложение обращается к `/api/workout-plans` и `/api/workout-plans/search`.
5. Backend возвращает актуальные данные.

## UC-06. Добавление записи прогресса
**Актор:** пользователь  
**Предусловия:** пользователь авторизован  
**Результат:** новая запись отображается в истории

**Основной поток:**
1. Пользователь вводит дату, вес и заметку.
2. Приложение отправляет запрос `POST /api/progress`.
3. Backend сохраняет запись.
4. Mobile обновляет список и график прогресса.

## UC-07. Просмотр административного списка пользователей
**Актор:** администратор  
**Предусловия:** выполнен вход под ролью `ADMIN`  
**Результат:** администратор видит список пользователей

**Основной поток:**
1. Администратор открывает служебный раздел.
2. Приложение отправляет запрос `GET /api/admin/users`.
3. Backend проверяет роль.
4. Backend возвращает список пользователей.

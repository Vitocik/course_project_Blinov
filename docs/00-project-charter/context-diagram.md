# Контекстная диаграмма

## Граница системы
Система — приложение **Nutrition Coach**.  
Снаружи находятся пользователь, администратор и инфраструктура запуска.

## Внешние акторы
- **Пользователь** — регистрируется, авторизуется, ведёт питание, тренировки и прогресс.
- **Администратор** — просматривает список пользователей и служебную информацию.
- **Мобильное устройство** — отображает интерфейс и сохраняет локальный кэш.
- **Сервер приложений** — обрабатывает запросы REST API.
- **База данных** — хранит учётные записи, профили, блюда, тренировки и записи прогресса.

## Основные потоки взаимодействия
1. Пользователь вводит данные в mobile-клиент.
2. Mobile обращается к backend по REST API.
3. Backend проверяет токен, выполняет бизнес-логику и обращается к БД.
4. Система возвращает данные обратно в mobile.
5. Для части сценариев mobile использует локальный кэш и может работать без постоянного соединения.

## Результат
Контекстная схема показывает, что проект является клиент-серверной системой с чётким разделением ролей и ответственностей.
